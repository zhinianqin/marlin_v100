# 2026-05-11 SM70 kU4 Zero-Point Prototype Plan

## 背景

当前 dense `kU4B8` 原型已经通过 CUTLASS SM70 threadblock mainloop 跑通：

- `CTA=128x128x32`
- `Warp=64x64x32`
- `warps=4`
- `stage=2`
- Volta TensorOp `8x8x4`
- A operand 使用 CUTLASS `IteratorA -> SmemIteratorA`
- B operand 使用自定义 iterator：packed int4 global load -> register dequant -> scale multiply -> CUTLASS predefined B SMEM layout -> MMA

已验证的 B shared-memory layout 是：

```text
RowMajorVoltaTensorOpMultiplicandBCongruous<16>
```

本阶段目标是在不改变 Torch op schema、不改变 GEMM mainloop 框架的前提下，开发 dense `kU4` integer zero-point 原型。`kU4B8` 的数学语义是：

```text
B = (q - 8) * scale
```

`kU4` 的数学语义是：

```text
B = (q - zp) * scale
```

因此第一阶段只应把额外复杂度限制在 B iterator 的 zero-point 注入路径里，不应同时改变 CTA tile、warp tile、A path、B SMEM layout 或 epilogue。

## 原型目标

- dense `marlin_gemm` 支持 `kU4` + integer zero-point。
- 继续保持 `marlin_gemm` Torch op schema、Python wrapper 参数和调用方式不变。
- 继续只支持 SM70 / Tesla V100。
- 继续只支持 fp16 A、fp16 C、fp16 scales。
- 暂不支持 act-order、float zero-point、bias、global scale、atomic-add。
- B operand 继续写入 CUTLASS predefined `RowMajorVoltaTensorOpMultiplicandBCongruous<16>` shared-memory layout。
- qweight repack 第一版复用当前 SM70 native `16x64` tile layout。
- zero-point 第一版使用 packed row-major `[num_groups, size_n / 8]`，pack order 与 qweight 的 int4 native path 一致。

## 不做的事情

- 不调用 `cutlass::gemm::device::Gemm`。
- 不恢复 `TransformB`。
- 不恢复 `BColumnMajorCrosswise` 或 dense B proxy 转置路径。
- 不把 packed B 先写普通 row-major shared memory 再二次搬运。
- 不在 repack 阶段预计算 `q - zp`，因为结果范围是 `[-15, 15]`，4-bit 无法无损表达。
- 不在本阶段扩展 MoE。
- 不在本阶段实现 `kU8B128`。

### 3. 在 IteratorB 注入 zero-point

`Sm70U4ZpIteratorB` 在 `Sm70U4B8IteratorB` 基础上增加：

```cpp
uint32_t const* qzeros_;
mutable int cached_zero_group_;
mutable half2 cached_zeros_[ThreadMap::Iterations::kContiguous * 4];
```

zero-point 地址语义：

```text
qzeros shape = [num_groups, size_n / 8]
zero_offset = group * (size_n / 8) + logical_n / 8
```

约束：

- `logical_n` 必须按 8-column 对齐。
- zero-point pack order 与 qweight int4 native pack order 一致。
- `group_size` 支持 `32, 64, 128`，`-1` 可作为 full-K 单 group 路径保留。

### 4. 第一版 correctness dequant

第一版先使用最直接的 half2 数学路径，优先保证正确性：

```cpp
half2 q01_23[2];
half2 q45_67[2];
half2 z01_23[2];
half2 z45_67[2];

marlin::dequant<half2, vllm::kU4.id(), false>(
    static_cast<int>(qword), q01_23);
marlin::dequant<half2, vllm::kU4.id(), false>(
    static_cast<int>(qword >> 8), q45_67);

marlin::dequant<half2, vllm::kU4.id(), false>(
    static_cast<int>(zword), z01_23);
marlin::dequant<half2, vllm::kU4.id(), false>(
    static_cast<int>(zword >> 8), z45_67);

half2* frag_vec = reinterpret_cast<half2*>(frag.data() + frag_base);
frag_vec[0] = __hmul2(__hsub2(q01_23[0], z01_23[0]), scale_vec[0]);
frag_vec[1] = __hmul2(__hsub2(q01_23[1], z01_23[1]), scale_vec[1]);
frag_vec[2] = __hmul2(__hsub2(q45_67[0], z45_67[0]), scale_vec[2]);
frag_vec[3] = __hmul2(__hsub2(q45_67[1], z45_67[1]), scale_vec[3]);
```

说明：

- 这条路径不是最终性能上限。
- 它能最快建立 correctness baseline。
- 后续优化以 SASS、register、spill 和 TFLOPs 结果为准。

### 5. zero-point cache 优化

第一版正确性通过后，立即合并 scale cache 与 zero cache：

```cpp
refresh_metadata_cache(group)
```

设计方向：

- scale cache 和 zero cache 按 N contiguous access 缓存。
- 不按 K 维缓存 zero，避免寄存器数量失控。
- `group_size=-1/64/128` 中，一个 group 会跨多个 K tile，cache 复用收益较大。
- `group_size=32` 中，每个 CTA_K 基本换 group，但同一个 iterator load 内仍可减少重复 global load / dequant。

需要对比两种 cache 形式：

```text
方案 A：缓存 dequant 后的 half2 zero
方案 B：缓存 packed uint32 zero，使用时再 dequant
```

决策指标：

- median latency
- TFLOPs
- ptxas registers
- spill stores / spill loads
- SASS 中 zero dequant 指令数

## repack 策略

### qweight

第一版不修改 qweight repack。

当前 `awq_marlin_repack` 和 `gptq_marlin_repack` 的 int4 non-A8 输出已经是 SM70 native `16x64` tile：

```text
tile shape = 16x64
word_offset = local_k * 8 + local_n_vec
每个 uint32 = 同一 K 行、连续 8 个 N 列
pack order = {0, 2, 4, 6, 1, 3, 5, 7}
```

这个布局对 `kU4B8` 和 `kU4` 都适用。

### qzeros

第一版使用现有测试 helper 的 zero-point pack 语义：

```text
shape = [num_groups, size_n / 8]
pack order = {0, 2, 4, 6, 1, 3, 5, 7}
```

原因：

- zero-point 每个 group/N-vector 只需读取一次，数据量远小于 qweight。
- 先保持 qzeros row-major，便于验证数学语义。
- 如果 benchmark 证明 zero load 或 zero dequant 是主要瓶颈，再新增 qzeros preprocess/repack。

后续可选优化：

- 将 qzeros 预排成和 `Sm70U4ZpIteratorB` N access 完全一致的 tile-major layout。
- 将 qzeros 与 scales 的 N 向量读取模式统一，减少 address arithmetic。
- 对 `group_size=128` 特化 qzeros group offset，降低整数除法/乘法压力。

## host dispatch 计划

修改：

```text
csrc/quantization/marlin/marlin.cu
```

当前 dense 只允许：

```cpp
b_type == vllm::kU4B8
```

计划改为：

```cpp
b_type == vllm::kU4B8 || b_type == vllm::kU4
```

dispatch 逻辑：

```cpp
if (b_type == vllm::kU4) {
  TORCH_CHECK(has_zp,
              "SM70 uint4 dense path requires integer zero-points.");
  TORCH_CHECK(!is_zp_float,
              "SM70 uint4 dense path does not support float zero-points.");
  return sm70_marlin_u4_gemm(a, c, b_q_weight, b_scales, b_zeros,
                             size_m, size_n, size_k, group_size);
}

if (b_type == vllm::kU4B8) {
  TORCH_CHECK(!has_zp,
              "SM70 uint4b8 dense path does not support zero-points.");
  return sm70_marlin_u4b8_gemm(a, c, b_q_weight, b_scales,
                               size_m, size_n, size_k, group_size);
}
```

保留现有限制：

- `!has_act_order`
- `!has_bias`
- `global_scale.numel() == 0`
- `!use_atomic_add`
- `is_k_full`
- `size_k % 32 == 0`
- `size_n % 64 == 0`

## 优化实验顺序

### v0 direct zero load

每个 qweight qword 对应读取一次 zword，直接 dequant zero 并计算：

```text
(q - zp) * scale
```

目标：

- correctness baseline
- 明确寄存器和指令成本

### v1 zero half2 cache

缓存 dequant 后的 zero half2：

```text
cached_zeros_[c * 4 + 0..3]
```

目标：

- 减少重复 zero global load
- 减少重复 zero dequant
- 观察 register 是否接近 255 或出现 spill

### v2 packed zero cache

只缓存 packed `uint32_t zword`：

```text
cached_zero_words_[c]
```

目标：

- 降低寄存器占用
- 接受更多 zero dequant 指令
- 通过 benchmark 判断是否比 v1 更优

### v3 integer subtract probe

尝试在 integer domain 处理 `q - zp`，再转换到 half2。

注意：

- 必须保持 signed result 的数学语义。
- 只在 SASS 指令数明显下降、正确性稳定、无 spill 时保留。
- 不允许引入难以验证的 packed saturation 语义。

### v4 qweight/qzero offset pressure

对比：

```text
预计算 qweight_offsets_ / qzero_offsets_
动态计算 qweight offset / qzero offset
半预计算：只预计算 N base，K advance 动态叠加
```

目标：

- 在 address arithmetic 和 register pressure 之间找平衡。
- kU4 比 kU4B8 多 zero metadata，可能不能照搬 kU4B8 的 offset 预计算策略。
