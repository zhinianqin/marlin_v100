# 2026-05-11 SM70 kU4 Zero-Point Prototype Result

## 改动内容

- dense `kU4` integer zero-point 原型接入 `marlin_gemm`。
- 继续保持公共 Torch op schema 和 Python wrapper 调用参数不变。
- 不调用 `cutlass::gemm::device::Gemm`，继续使用 CUTLASS SM70 threadblock 组件：
  - `DefaultMmaCore`
  - `MmaPipelined`
  - Volta TensorOp `8x8x4`
  - `DefaultEpilogueVoltaTensorOp`
- B operand 保持纯正路径：
  - packed int4 global load
  - register streaming zero-point dequant
  - fp16 scale multiply
  - 写入 CUTLASS predefined Volta B-congruous shared memory layout
  - MMA 从 shared memory 读取 B
- B shared-memory layout 继续通过 `static_assert` 固定为：

```text
RowMajorVoltaTensorOpMultiplicandBCongruous<16>
```

- 不恢复 `BColumnMajorCrosswise`。
- 不恢复 `ExplicitTransformB`。
- 不引入 global half B 中间结果。

## 最终 kernel 配置

最终保留 true 8-warp 版本：

```cpp
constexpr int kCtaM = 128;
constexpr int kCtaN = 128;
constexpr int kCtaK = 32;
constexpr int kWarps = 8;
constexpr int kThreads = kWarps * 32;

using ThreadblockShape = cutlass::gemm::GemmShape<128, 128, 32>;
using WarpShape = cutlass::gemm::GemmShape<64, 32, 32>;
```

关键点：只把 `kWarps` 从 `4` 改到 `8` 不够。CUTLASS threadblock MMA 的真实 warp 数来自 `ThreadblockShape / WarpShape`，如果仍保留 `WarpShape=64x64x32`，`128x128x32` CTA 实际仍只有 4 个 MMA warp，多出来的 launch threads 只是空闲线程。

因此本轮增加了约束：

```cpp
static_assert(MmaCore::kThreads == kThreads,
              "SM70 kU4 launch threads must match CUTLASS warp count.");
```

这样可以防止后续误把 launch block 调到 256 threads，却没有真正改变 CUTLASS MMA warp mapping。

### 性能结论

- true 8-warp 版本达成了本轮首要目标：消除 kU4 `group_size=128/64/32` spill。
- 性能仍很低：kU4 zero-point 当前只有 `35.32 TFLOPs kernel_like`，约为 V100 `125 TFLOPs` 理论峰值的 `28.26%`。
- 同 shape 下 kU4B8 `kernel_like` 为 `76.85 TFLOPs`，说明主 GEMM 框架和 benchmark 路径本身仍可达到较高吞吐。
- pure B shared-memory matmul baseline 之前记录为 `91.98 TFLOPs`。kU4 zero-point 距离 `90+ TFLOPs` 目标仍有很大差距。
- 当前瓶颈应继续集中在 `Sm70U4ZpIteratorB::load()`，而不是扩展格式或进入 MoE。

## 已知瓶颈

- kU4 zero-point 比 kU4B8 多一条 packed zero-point load/dequant 路径。
- 当前 `IteratorB::load()` 同时处理 qweight、qzero、scale、half2 sub/mul，虽然没有 spill，但指令数和依赖链仍很重。
- true 8-warp 通过缩小 warp N tile 消除了 spill，但每个 warp 的 N 方向工作量更小，可能降低了局部数据复用和 issue 效率。
- `128x256x32/8` 能稍微提高 `kernel_like`，但引入大量 spill，暂时不保留。

## 下一步

- 不扩展 MoE，不扩展 `kU8B128`，先继续优化 dense kU4 zero-point。
- 优先优化 `Sm70U4ZpIteratorB::load()`：
  - 缩短 qword、zword、scale、half2 临时变量生命周期。
  - 拆分 zero-point dequant 和 scale multiply 的顺序，降低活跃寄存器峰值。
  - 检查 SASS 中 qweight/qzero load 是否稳定合并访存。
  - 评估把 zero-point repack 成更贴近 B iterator 访问顺序的布局，减少地址计算。
  - 对比缓存 packed zero word 和缓存 dequant 后 half2 zero 的指令数与寄存器压力。
- 若能在无 spill 前提下压低 IteratorB 指令数，再重新尝试更大的 CTA N tile 或新的 warp mapping。
