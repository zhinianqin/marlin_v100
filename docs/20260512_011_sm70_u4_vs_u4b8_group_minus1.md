# 2026-05-12 SM70 u4 与 u4b8 group_size=-1 实现差异结论

## 背景

当前 dense `uint4` zero-point 路径中，`group_size=-1` 的性能明显高于
`group_size=128`：

| quant | group_size | kernel_like TFLOPs |
| --- | ---: | ---: |
| `uint4` | `-1` | 79.01 |
| `uint4` | `128` | 72.82 |

而 dense `uint4b8` 在原实现中表现相反，`group_size=-1` 明显低于
`group_size=128`：

| quant | group_size | kernel_like TFLOPs |
| --- | ---: | ---: |
| `uint4b8` | `-1` | 69.62 |
| `uint4b8` | `128` | 75.28 |

本结论用于解释为什么 `uint4 group_size=-1` 能快很多，以及为什么这个直觉不能直接套到
`uint4b8 group_size=-1`。

## uint4 group_size=-1 的实现

`uint4` zero-point kernel 在 host 侧会按 shape 静态分裂 full/residue 路径：

```cpp
bool const full_tile = (size_k % kCtaK == 0 && size_n % kCtaN == 0);
```

典型 benchmark 形状 `5120x4096x4096` 满足 full-tile 条件，因此
`group_size=-1` 会走静态实例：

```cpp
launch_sm70_marlin_u4_gemm<-1, true>(...)
```

也就是说 `GroupSize=-1` 和 `FullTile=true` 都是编译期模板参数，不是运行时分支。

在 `Sm70U4ZpIteratorB::scale_group()` 中，`group_size=-1` 直接映射为单一
group：

```cpp
if constexpr (kGroupSize == -1) {
  return 0;
} else {
  return logical_k / kGroupSize;
}
```

`load()` 每次会根据当前 K 位置调用 `refresh_metadata_cache(current_group)`。
但对于 `group_size=-1`，`current_group` 永远是 `0`，所以只有第一次需要刷新：

```cpp
if (cached_group_ == group) {
  return;
}
```

刷新 metadata 时，kernel 会一次性缓存：

- `cached_scales_`
- dequant 后的 zero-point
- `cached_bias_ = -zero * scale`

之后 hot path 只需要流式加载 qweight、dequant，然后执行：

```cpp
frag = __hfma2(q, scale, bias);
```

也就是把：

```text
(q - zero) * scale
```

变成：

```text
q * scale + (-zero * scale)
```

zero dequant 和 `zero * scale` 不再在每个 qweight access 中重复执行。

## 为什么 uint4 group_size=-1 比 128 快

`uint4 group_size=128` 的 group 计算是：

```cpp
logical_k / 128
```

当前 CTA K 是 `32`，因此每 4 个 K tile 切换一次 metadata group。对于
`K=4096`，每个 CTA_N 会经历 32 个 group。每次 group 变化都会重新执行：

- scale load
- qzero load
- qzero dequant
- `bias = -zero * scale`
- qzero 地址计算
- scale 地址计算
- `cached_group_` 更新

而 `group_size=-1` 在整条 K 维只刷新一次 metadata。qweight 主路径基本相同，
但 metadata 路径少了大量工作。

资源结果也支持这个结论：

| specialization | REG | STACK | LOCAL |
| --- | ---: | ---: | ---: |
| `uint4 <-1, true>` | 246 | 0 | 0 |
| `uint4 <128, true>` | 255 | 0 | 0 |

`group_size=-1` 的 full-tile 实例寄存器压力更低，`group_size=128` 顶到 255
regs。SM70 上这个差异会影响调度余量和编译器安排。

因此 `uint4 group_size=-1` 快的原因是叠加的：

- metadata 只刷新一次
- zero dequant 和 bias 预计算只做一次
- hot path 使用 `hfma(q, scale, bias)`
- full-tile 静态实例寄存器更少

## uint4b8 的不同点

`uint4b8` 没有 zero-point tensor，也没有 `cached_bias_`。它的 hot path 是：

```text
deq(q_int4_bias8) * scale
```

对应实现是：

```cpp
marlin::dequant<half2, vllm::kU4B8.id(), false>(qword, deq);
frag = __hmul2(deq, scale);
```

因此 `uint4b8 group_size=-1` 虽然同样语义上只有一个 scale group，但它不能获得
`uint4` 中那部分 zero/bias cache 收益。

当前 `uint4b8` 原实现还有两个关键差异：

- 没有 full-tile 模板分裂，`load()` hot path 中始终保留 residue 边界判断。
- `GroupSize=-1` 被编译为常量 `group=0` 的独立路径，但实测这条 codegen 路径
  比 `group_size=128` cadence 更慢。

此前诊断实验已经证明，`uint4b8 group_size=-1` 的慢不是 qweight repack 主布局导致：

```text
max_diff = 0.0
group_size=-1 原路径: 2513.41 us, 68.35 TFLOPs
same qweight + repeated scales + 128 路径: 2319.36 us, 74.07 TFLOPs
```

同一份 `q_weight` 只要强制走 `group_size=128` kernel 就能明显变快，说明瓶颈在
`GroupSize=-1` 的 iterator / scale policy / codegen 路径，而不是 qweight 数据布局。

## 对 uint4b8 优化的启发

不要把 `uint4` 的 “group 永远为 0 就一定最快” 直觉直接套到 `uint4b8`。

`uint4` 中，`group_size=-1` 快是因为少做了大量 zero-point metadata 工作，并且
`cached_bias_` 把 hot path 压成了 `hfma`。`uint4b8` 没有 zero-point，也没有 bias
预计算，因此性能更容易被 scale cache 形态、边界分支和编译器 codegen 主导。

对 `uint4b8 group_size=-1`，更好的方向是：

- 语义仍保持单 scale group。
- 不扩展公共接口，不复制 scale metadata。
- kernel 内部采用虚拟 `128` refresh cadence。
- cache key 按 `logical_k / 128` 变化。
- scale row 实际仍映射到第 0 行，或使用等价的 `scale_group_stride=0` policy。

临时实验结果已经验证该方向有效：

| quant | group_size | 原 kernel_like TFLOPs | 虚拟 128 policy TFLOPs |
| --- | ---: | ---: | ---: |
| `uint4b8` | `-1` | 69.62 | 75.00 |
| `uint4b8` | `128` | 75.28 | 75.20 |

也就是说，`uint4b8 group_size=-1` 应该做成“语义单 group、实现上虚拟 128 cadence”
的专门路径，而不是继续依赖当前常量 `group=0` 的 codegen。

## 下一步建议

- 正式实现时先只改 `uint4b8 group_size=-1` 的 scale policy，不动 qweight repack。
- 保持 `uint4b8 group_size=64/128` 当前路径不变，避免影响已有最快路径。
- `uint4b8 group_size=32` 的 direct-scale 与 cache-per-CTA-K 需要单独 A/B，不要和
  `-1` 的虚拟 128 policy 绑在同一笔改动里。
- 暂不把 `uint4b8` full-tile 模板分裂作为第一优先级，因为实验中 full-tile 实例
  增加到 255 regs，没有资源收益。
