# SM70 u4 256N / 8-Warp 对齐 u4b8

## 摘要

本次改动把 dense SM70 `uint4` scale+bias 路径从 `128x128x32 / 4 warp`
切到 `128x256x32 / 8 warp`，并以
`csrc/quantization/marlin/sm70_marlin_u4b8_gemm.cu` 的当前实现形态作为结构标准。

保留不变的内容：

- Torch op schema 和 Python wrapper 不变
- `b_q_weight` repack layout 不变
- `b_scales` / `b_zp_bias` shape、dtype、语义不变
- dense `uint4` 仍然只使用 scale+bias metadata，不恢复 packed zero-point 输入
- 不保留旧 `128x128x32 / 4 warp` fallback

允许保留的差异只有 `uint4` 本身需要的 scale+bias metadata、`__hfma2(q, scale, bias)`
数学、`b_zp_bias` 参数，以及 `u4` 相关类名 / kernel 名 / trait 名。

## 几何变化

`csrc/quantization/marlin/sm70_marlin_u4_gemm.cu` 现在使用：

| 常量 | 当前值 |
|---|---:|
| `kCtaM` | `128` |
| `kCtaN` | `256` |
| `kCtaK` | `32` |
| `kWarps` | `8` |
| `kThreads` | `256` |
| `WarpShape` | `64x64x32` |

`FullTile=true` 条件随之变成：

```cpp
size_k % 32 == 0 && size_n % 256 == 0
```

因此 `size_n=128` 等旧 128N 能整除、但不能被 256 整除的场景现在走 residue
实例。这是本轮统一单路径后的预期行为。

## IteratorB 对齐

`Sm70U4ZpBiasIteratorB` 已按 `Sm70U4B8IteratorB` 对齐：

- `ThreadMap::Iterations::kStrided == 1`
- `ThreadMap::Iterations::kContiguous == 4`
- `ThreadMap::Delta::kContiguous == 64`
- 删除原来的 `qweight_offsets_[ThreadMap::Iterations::kCount]`
- 使用 `qweight_base_offset_`
- `operator++()` 只更新 `qweight_base_offset_`
- full-tile qweight load 使用一次 `uint4` 读取 4 个 64-column qword
- residue qweight load 使用 `qweight_offset(c)` 做 scalar 读取，并对 invalid fragment 写 0

full/residue 热路径仍保持流式 dequant 顺序：

1. 读取 packed qword
2. low dequant
3. 立刻 `__hfma2(q, scale, bias)` 写 `frag[0..1]`
4. high dequant
5. 立刻写 `frag[2..3]`

positive group 的 metadata 仍在 `load()` 中、`load_full_tile()` /
`load_residue_tile()` 外 cache。full/residue 内层只消费 `cached_scales_` 和
`cached_bias_`，不直接读取 metadata。

## 与 u4b8 的必要差异

人工 diff 检查命令：

```bash
diff -u \
  csrc/quantization/marlin/sm70_marlin_u4b8_gemm.cu \
  csrc/quantization/marlin/sm70_marlin_u4_gemm.cu
```

剩余差异属于以下范围：

- `Sm70U4B8IteratorB` vs `Sm70U4ZpBiasIteratorB`
- `Sm70U4B8GemmTraits` vs `Sm70U4ZpBiasGemmTraits`
- `sm70_marlin_u4b8_gemm_kernel` vs `sm70_marlin_u4_gemm_kernel`
- `u4` 额外持有 `zp_bias_` 和 `cached_bias_`
- `u4` constructor / `cache_current_group_metadata()` 同时 cache scale 和 bias
- `u4` 使用 `vllm::kU4` 和 `__hfma2(q, scale, bias)`
- `u4b8` 使用 `vllm::kU4B8` 和 `__hmul2(q, scale)`
- `u4` launcher / kernel 多传一个 `b_zp_bias`

没有保留旧 `128N` 几何、旧 `qweight_offsets_[]` 模型或 packed zero-point dense
路径。

## 性能判断

这次 256N/8-warp 的收益主要来自：

- full-tile qweight 由 4 个分散 scalar access 变成一次 `uint4` vector load。
- `qweight_base_offset_ + c` 模型减少了 fragment access 的地址状态。
- `kWarps=8` 和 pure GEMM sweep 中较优的 256N 形态一致，full-tile 主路径没有 spill。

positive group 的提升小于 `group_size=-1`，原因是 `u4` 比 `u4b8` 额外需要
`b_zp_bias` 平面和 `cached_bias_`，并使用 `HFMA2` 而不是单纯 `HMUL2`。
residue positive group 已经顶到 `REG=255` 并产生小 spill，因此如果后续要继续压榨
非 256N 边界场景，需要单独收缩 residue metadata live range 或给 residue 做更轻的
metadata cache 策略。

## 后续方向

- 针对 `u4<32/64/128,false>` residue 实例，减少 valid 分支、zero 写入和 scale+bias
  cache 的同时活跃区间，目标是消除 `12B/12B` spill。
- 检查 full-tile `u4<32/64/128,true>` 的 SASS，确认 scale/bias 平面是否形成理想的
  连续 half2 load；如果没有，可以考虑更明确的 vectorized metadata load。
- 对 `size_n=128` 等小 N 场景评估是否需要单独的小 N kernel，但本轮不恢复旧双路径。
