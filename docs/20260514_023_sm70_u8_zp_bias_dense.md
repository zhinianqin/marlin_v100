# SM70 dense uint8 scale+bias GEMM

## 背景

本轮新增 dense `uint8` / `vllm::kU8` 支持，语义对齐当前 dense
`uint4` scale+bias 路径：运行时输入两个 fp16 metadata 平面：

- `b_scales[num_groups, N]`
- `b_zp_bias[num_groups, N]`

kernel 内执行：

```text
dequant = q * scale + zp_bias
zp_bias = -zero_point * scale
```

这里的 `uint8` 是 zero-point 量化路径，不是旧的 `uint8b128`。本轮不复用
`uint8b128` 名称，也不改 MoE 8-bit kernel；Python 支持矩阵里暂时移除
`uint8b128` 的 dense/MoE 正确性支持，只保证现有 MoE/`uint8b128` 编译通过。

## 实现范围

新增文件：

- `csrc/quantization/marlin/sm70_marlin_u8_gemm.cu`

Dense dispatch 更新：

- `marlin.cu` 允许 SM70 dense `kU4 / kU4B8 / kU8`。
- `kU8` 和 `kU4` 一样要求 fp16 `b_zp_bias`，并且 `use_zp_bias=true`。
- `kU8` 拒绝 packed integer zero-point。
- `kU8` 要求 `size_k % 32 == 0`、`size_n % 64 == 0`、无 act-order。
- `kU8B128` 仍不进入 SM70 dense custom kernel。

Python/helper 更新：

- 新增 `scalar_types.uint8 = ScalarType(0, 8, False, 0)`。
- 新增 `marlin_quantize_uint8_zp_bias(...)`。
- 新增 `marlin_dequantize_uint8_zp_bias(...)`。
- dense benchmark 支持 `--quant-types uint8`。
- `calibration.py` 中 `uint8` 为 dense supported、MoE unsupported；`uint8b128` 暂时 dense/MoE 都 unsupported。

## 几何与对齐

`sm70_marlin_u8_gemm.cu` 以当前 `sm70_marlin_u4_gemm.cu` 为结构标准：

| 项目 | 值 |
|---|---:|
| `kCtaM` | 128 |
| `kCtaN` | 256 |
| `kCtaK` | 32 |
| `kWarps` | 8 |
| `kThreads` | 256 |
| `WarpShape` | `64x64x32` |

IteratorB 的 metadata 策略也与 u4 对齐：

- `GroupSize=-1`：constructor 内联 cache `scale + zp_bias`。
- `GroupSize=32/64/128`：在 `load()` 内、`load_full_tile()` /
  `load_residue_tile()` 外刷新当前 group 的 metadata cache。
- full/residue 热路径只消费 `cached_scales_` / `cached_bias_`。

与 u4 的必要差异：

- `kU8ValuesPerWord = 4`，每个 8-element access 需要两个 `uint32` qweight word。
- full-tile 每次读取两个 `uint4`，第一个覆盖每个 N64 subtile 的前 4 个 uint8，
  第二个覆盖后 4 个 uint8。
- dequant 使用 `vllm::kU8`。

## qweight macro-N layout

8-bit 使用与 u4 相同的 macro-N 思路，但 pack factor 从 8 变为 4：

```text
pack_factor = 4
kU8ValuesPerWord = 4
kU8WordsPerTile = 16 * 64 / 4 = 256

layout = [k_tile][n256_macro][local_word][n64_subtile]

local_word = local_k * 16 + local_n_word
local_n_word = local_n / 4
```

完整 offset：

```text
k_tile = logical_k / 16
local_k = logical_k % 16
n_tile = logical_n / 64
macro_n_tile = n_tile / 4
macro_first_n_tile = macro_n_tile * 4
subtile = n_tile - macro_first_n_tile
subtile_count = min(4, n_tiles - macro_first_n_tile)
local_n_word = (logical_n % 64) / 4
local_word = local_k * 16 + local_n_word

offset =
  k_tile * (size_n * 4) +
  macro_n_tile * 4 * 256 +
  local_word * subtile_count +
  subtile
```

`gptq_marlin_repack.cu`、`awq_marlin_repack.cu` 和 Python reference
均同步使用该 8-bit macro-N layout。`is_a_8bit=true` 的 legacy 路径没有纳入本轮重排。

## Benchmark

命令：

```bash
DENSE_ARGS="--models ideal --batch-sizes 5120 --quant-types uint8 --group-sizes -1 32 64 128 --act-order off --is-k-full true --report-tflops --warmup-iters 10 --iters 30" \
  BENCH_PRESET=quick ./benchmark.sh dense
```

日志：

```text
benchmarks/results/20260514_211300_dense_quick.log
```

`5120x4096x4096`，`uint8`：

| group_size | operator TFLOPs | kernel_like TFLOPs |
|---:|---:|---:|
| `-1` | 81.17 | 82.50 |
| `32` | 70.51 | 72.01 |
| `64` | 72.83 | 72.47 |
| `128` | 71.14 | 71.85 |

观察：

- `GroupSize=-1` 表现接近当前 u4/u4b8 的 80 TFLOPs 级别。
- positive groups 明显更低，主要原因是每个 access 需要两份 qweight word，
  同时要加载 `scale + zp_bias` 两个 metadata 平面；相比 u4 的 4-bit unpack，
  8-bit 路径算术更轻，metadata 和 load 压力占比更高。
- full-tile 已经无 spill，因此 positive group 主要需要继续看 metadata load
  合并、cache reuse 或 `scale+bias` 向量化访存策略，而不是先处理 spill。

## 后续方向

- 将 residue kernel 继续细分为 `ResidueNOnly / ResidueKOnly / ResidueKAndN`，
  用 `if constexpr` 删除不需要的边界变量，目标消除 residue positive group spill。
- 对 positive group full-tile 研究 `scale + zp_bias` 的 128-bit vector load 或更紧的 cache
  布局，减少 metadata 带宽和地址计算。
- 单独重做 `uint8b128` / MoE 8-bit 支持，不复用本轮 dense `uint8` zero-point 语义。
