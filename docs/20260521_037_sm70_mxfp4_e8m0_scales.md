# SM70 Dense MXFP4 E8M0 Scales 语义对齐实验

## 摘要

- Baseline：`bc6feec47565746a6052cc5bc6ba2387dbe19bcc`
- After：同一 HEAD 上的工作区改动，未提交
- 目标：将 SM70 dense MXFP4 的 `b_scales` 从 preconverted `float16` 改为原版 Marlin 语义的 raw `torch.float8_e8m0fnu`
- Kernel 行为：metadata cache 阶段读取 E8M0 bytes，dequant 为 `half2` 后写入 `cached_scales_`，`load_macro_n_aligned()` 继续使用缓存后的 fp16 scale
- 结论：15 个 CTA 中无明确衰退、无轻微退化；13 个 CTA 改善，2 个基本持平；所有目标 kernel 均无 stack/local/spill

## 代码改动

- `marlin.cu`
  - `float4_e2m1f + float8_e4m3fn` 继续路由到 NVFP4，要求 `global_scale`
  - `float4_e2m1f + float8_e8m0fnu` 改为路由到 MXFP4，禁止 `global_scale`
  - `float4_e2m1f + float16 scales` 不再作为 MXFP4 合法输入

- `sm70_marlin_mxfp4_gemm.cu`
  - `b_scales` 指针语义从 `half const*` 改为 `uint8_t const*`
  - 新增 E8M0 byte 到 fp16 bits 的 device helper
  - 删除 `cache_metadata_lane_vectors()` / `cache_metadata_vector_words()` 的 fp16 直读路径，改为 `cache_metadata_e8m0_scales()`
  - qweight layout、CTA dispatch、group_size=32 contract、`load_macro_n_aligned()` qweight dequant 主体保持不变

- `tests/helpers.py`
  - `marlin_quantize_mxfp4()` 现在返回 `torch.float8_e8m0fnu` scales
  - `marlin_dequantize_mxfp4()` 按 raw E8M0 scales 参考语义计算

## 结论

- MXFP4 已对齐为 raw `torch.float8_e8m0fnu` scales 输入语义，host routing 与原版 Marlin 的 `float8_e8m0fnu` MXFP4 判定一致。
- Kernel 内部将 E8M0 scale 在 metadata cache 阶段转换为 fp16 `half2`，没有改变 qweight dequant 主体，也没有引入 stack/local/spill。
- 本次同机同命令 benchmark 未观察到性能衰退；多数 CTA 的 `kernel_like_us` 有 1%-3% 改善。
- benchmark 结果只记录事实，不作为自动回滚依据；当前改动保留在工作区，未自动提交。
