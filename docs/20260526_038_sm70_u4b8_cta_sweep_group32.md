# 2026-05-26 SM70 U4B8 group_size=32 CTA sweep 结论

## 背景

本轮 benchmark 目标是 dense `uint4b8` 的 SM70 Marlin 路径：

```text
csrc/quantization/marlin/sm70_marlin_u4b8_gemm.cu
```

固定条件：

| 项 | 值 |
|---|---|
| branch | `refactor/cutlass-sm70-marlin-v100` |
| commit | `9fd793e` |
| GPU | `Tesla V100-SXM2-32GB` |
| capability | `sm70 (7.0)` |
| quant | `uint4b8` |
| group_size | `32` |
| model | `ideal` |
| M | `1, 2, 4, 8, 16, 32, 64` |
| K/N | `4096 / 4096` |
| act_order | `off` |
| is_k_full | `true` |
| use_fp32_reduce | `true` |
| warmup / iters | `20 / 100` |

命令口径：

```bash
COMMON_DENSE_ARGS="--models ideal --batch-sizes 1 2 4 8 16 32 64 --quant-types uint4b8 --group-sizes 32 --act-order off --is-k-full true --use-fp32-reduce true --warmup-iters 20 --iters 100 --report-tflops"

for cta in \
  32x128x4 32x256x4 \
  64x64x4 64x128x4 64x128x8 64x256x4 64x256x8 \
  128x64x4 128x64x8 128x128x4 128x128x8 128x256x8 \
  256x64x4 256x64x8 256x128x8
do
  SM70_MARLIN_U4B8_CTA="$cta" \
    BENCH_PRESET=quick \
    DENSE_ARGS="$COMMON_DENSE_ARGS" \
    ./benchmark.sh dense
done
```

15/15 个 CTA benchmark 均完成，没有 unsupported 或 runtime failure。

## kernel_like_us best

`kernel_like_us` 是主观察口径，复用预分配 output，更接近 kernel 本身耗时。

| M | best CTA | torch_us | marlin_us | speedup | torch_tflops | marlin_tflops | launch_dominated |
|---:|---|---:|---:|---:|---:|---:|---|
| 1 | `64x64x4` | 59.39 | 166.91 | 0.36x | 0.56 | 0.20 | yes |
| 2 | `64x64x4` | 79.87 | 166.91 | 0.48x | 0.84 | 0.40 | yes |
| 4 | `64x64x4` | 79.87 | 166.91 | 0.48x | 1.68 | 0.80 | yes |
| 8 | `64x64x4` | 79.87 | 167.94 | 0.48x | 3.36 | 1.60 | yes |
| 16 | `64x64x4` | 79.87 | 171.01 | 0.47x | 6.72 | 3.14 | yes |
| 32 | `64x64x4` | 68.61 | 173.06 | 0.40x | 15.65 | 6.20 | no |
| 64 | `32x128x4` | 73.73 | 178.16 | 0.41x | 29.13 | 12.05 | no |

结论：

- `64x64x4` 是本轮 small-M 的整体最佳，覆盖 `M=1..32` 的 best。
- `M=64` 时 `32x128x4` 最快，`kernel_like_us=178.16`，`marlin_tflops=12.05`。
- 相比默认 `128x256x8`，小 M 延迟从约 `278..313 us` 降到约 `167..178 us`。
- 这轮所有 Marlin best 仍慢于同日志中的 `torch.mm` baseline。

观察：

- `CTA_N=64` 通常更适合 small-M，尤其 `64x64x4` 和 `128x64x8`。
- `CTA_N=256` 在这组 small-M shape 中普遍偏慢。
- 默认 `128x256x8` 对 `M<=64` 不合适，尤其 `M=1` 和 `M=32` 都超过
  `300 us`。
- `Warps=8` 并不自动改善 small-M；很多 8-warp CTA 比 4-warp CTA 更慢。

## 后续建议

- 如果要优化 `uint4b8 group_size=32` 的 small-M 默认行为，优先考虑把
  `SM70_MARLIN_U4B8_CTA=64x64x4` 作为 `M<=32` 的候选策略。
- `M=64` 可以单独考虑 `32x128x4`。
- 不建议继续用默认 `128x256x8` 作为 small-M 的性能判断基线；它更像大 M 默认
  tile，在 `M<=64` 下固定开销过重。
- `M<=16` 属于 launch-dominated 区间，评估时应优先看 latency，不应把 TFLOPs
  当作吞吐上限。
